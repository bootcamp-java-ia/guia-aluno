# Semana 12 – Projeto Final e Demo Day

## 🎯 Big Picture
Entrega de valor supera código perfeito.
Esta semana **não é sobre aprender algo novo**.
É sobre **consolidar, finalizar e explicar com clareza** o que foi construído.

No mercado real, projetos não acabam quando o código fica "bonito",
mas quando **funcionam, entregam valor e respeitam o prazo**.

---

## 🧩 Conexão com o Projeto Final
Esta semana **é** o Projeto Final.
Tudo que veio antes converge aqui: ambiente, sintaxe, fluxo, coleções, POO,
clean code, exceções, persistência, CRUD e API.

> Não é obrigatório usar **tudo** — é obrigatório **dominar o que usou**.

---

## 🚀 MVP da Semana (Obrigatório)
Cada squad deve entregar:

- Sistema **funcional**, rodando do início ao fim
- Código versionado no `projeto-final`
- README claro explicando:
    - problema resolvido
    - solução adotada
    - como rodar
    - decisões técnicas principais
- Board do GitHub Project organizado
- Apresentação no Demo Day (10–15 min)

---

## 🎮 Side Quest (Opcional)
Para squads que finalizaram o MVP com folga:

- Integrar com uma API externa simples
- Deploy básico (Render, Fly.io, Railway, VPS)
- Front-end mínimo (HTML que consome a API)
- Dockerfile

⚠️ **Não avance** se o MVP não está sólido.

---

## 🛠 Conteúdo Técnico Consolidado
- Arquitetura geral do sistema
- Fluxo de dados (entrada → regra → persistência → resposta)
- Organização do código em camadas
- Documentação para quem vai usar/avaliar

Nenhum conceito novo — só **refinamento**.

---

## 👥 Dinâmica de Trabalho
- Squads em call própria
- Papéis rotativos: codifica / revisa / documenta
- Instrutor e equipe de apoio em call aberta para revisão e debug

> Se algo quebrar, isso faz parte. Resolver juntos é o aprendizado.

---

## 🤖 Second Brain – Reforço Final
Use IA para: revisar README, identificar furos, gerar perguntas que a banca pode fazer.

❌ Não use IA para: "finalizar" o projeto por você ou gerar código que ninguém entende.

> **Se o squad não sabe explicar uma linha, essa linha não está pronta.**
